# AI Script Standard for Product Demo Videos

## Purpose

This document defines how product demo scripts should be written, interpreted, and refined.

It is designed so that any team member can write a rough draft in plain language, and an AI assistant can then convert that draft into a consistent, approved script style.

The goal is not to generate widely different creative outputs. The goal is to standardise structure, tone, and clarity across all demo videos.

---

## What the human should provide

The human does **not** need to write a polished script.

They only need to provide a rough draft or notes that include:

- what the product or solution is  
- what problem it solves  
- what the risk is without it  
- what the solution changes  
- what happens in the demo  
- what the audience should understand by the end  

This draft can be messy, incomplete, or written as rough paragraphs.

It does not need to follow the final wording or tone.

---

## What the AI must do

When given a rough script or notes, the AI must:

- preserve the intended meaning  
- improve structure and flow  
- rewrite the wording into the approved style  
- remove repetition  
- simplify jargon where possible  
- make the script sound natural when spoken aloud  
- ensure the script follows the standard storytelling format  
- avoid inventing missing product details  
- flag uncertainty instead of guessing  

The AI is not there to create a completely new concept. It is there to standardise and refine what has already been provided.

---

## Standard script structure

Unless explicitly told otherwise, all scripts should follow this structure:

1. Problem setup  
2. Negative user story  
3. Solution introduction  
4. Demo segment  
5. Under-the-hood explanation  
6. Positive user story  
7. Outcome / close  

Not every script needs all seven sections in full, but this is the default structure.

---

## Required tone

All scripts should be:

- clear for non-technical audiences  
- technically credible  
- visually understandable  
- concise  
- confident  
- calm rather than dramatic  
- natural to speak aloud  
- written in British English  

They should not sound like:

- marketing fluff  
- technical documentation  
- meeting notes  
- UI instructions being read out line by line  

---

## Writing style rules

### 1. Write for spoken delivery

Scripts must sound natural when read aloud.

Use:

- short to medium sentences  
- conversational phrasing  
- direct wording  

Avoid:

- overly long sentences  
- stacked clauses  
- wording that only works well on a slide, not in speech  

---

### 2. Be visual

The script should help the viewer imagine what is happening.

This is especially important in:

- negative user stories  
- positive user stories  
- demo segments  

Use concrete actions and visible outcomes.

---

### 3. Do not over-explain

The script should explain enough to build trust, but not so much that it becomes a lecture.

Keep deeper technical detail for:

- under-the-hood segments  
- supporting diagrams  
- follow-up content  

Do not overload the main narrative with implementation detail.

---

### 4. Keep user stories human and believable

Negative and positive user stories should describe realistic situations.

They should:

- follow a clear sequence  
- feel plausible  
- focus on what happens and why it matters  

They should not:

- become theatrical  
- anthropomorphise devices too heavily  
- drift into abstract technical explanation  

---

### 5. Introduce technical ideas in plain language

Where technical concepts are necessary, explain them in accessible terms.

Prefer:

- built-in trust mechanism  
- secure reference point  
- verified identity  
- authorised update  

Avoid unnecessary jargon unless it is essential and understood by the audience.

---

### 6. Match narration to visuals

If a script accompanies a UI walkthrough, animation, or diagram, the narration must broadly match what is visible.

Do not describe invisible background processes in detail during a visible product walkthrough unless that detail is necessary.

If technical depth is needed, place it in a separate under-the-hood segment.

---

### 7. Remove repetition

The AI must identify and remove duplicated meaning.

Do not restate the same point in slightly different ways unless repetition is deliberately used for emphasis.

---

### 8. Keep lists conversational

When rewriting slide bullets or note fragments into script, convert them into natural spoken sentences.

Example:

Instead of:
> Loss of EU market access. Regulatory scrutiny. Reputational damage.

Use:
> Loss of EU market access, regulatory scrutiny, and reputational damage.

---

## Standard section guidance

### Problem setup

This should explain the challenge clearly and quickly.

It should:

- frame the issue  
- establish relevance  
- avoid too much detail too early  

It should not:

- explain the whole solution  
- repeat later questions unnecessarily  

---

### Negative user story

This should show what goes wrong without the solution.

It should:

- feel visual and believable  
- show a realistic failure or attack path  
- make the risk concrete  

It should not:

- become too technical  
- feel melodramatic  
- overuse jargon  

---

### Solution introduction

This should explain what is being offered in simple terms.

It should:

- say what the product is  
- say what it is designed to do  
- position the company clearly  

It should not:

- imply uncertainty if the product should sound established  
- list all possible architectures unless strategically necessary  

---

### Demo segment

This should follow the product visually.

It should:

- match what is on screen  
- focus on visible actions and outcomes  
- support understanding  

It should not:

- read out every click  
- explain hidden system mechanics in detail  
- drift away from what the viewer can see  

---

### Under-the-hood segment

This is where the script can show engineering depth.

It should:

- explain the trust model or technical foundation simply  
- show why the solution is credible  
- justify complexity without becoming a lecture  

It should not:

- dump low-level implementation details  
- overload the viewer with terminology  

---

### Positive user story

This should show how the same kind of situation is handled correctly with the solution in place.

It should:

- feel calm, controlled, and practical  
- demonstrate ease of legitimate use  
- show the benefit clearly  

It should not:

- overstate  
- repeat the demo word-for-word unless intentional  

---

### Outcome / close

This should leave the audience with a clear takeaway.

It should:

- reinforce the value  
- simplify the result  
- end confidently  

It should not:

- introduce new concepts  
- feel bloated  
- repeat every benefit already stated  

---

## Convergence rules

The AI should reduce variation, not expand it.

When refining scripts, default to:

- one intro style  
- one narrative structure  
- one tone  
- one level of technical depth for the intended audience  

Do not generate multiple radically different versions unless asked.

The aim is consistency and reuse.

---

## What the AI must not do

The AI must not:

- invent product capabilities  
- claim compliance or certification without being told  
- add jargon for the sake of sounding technical  
- rewrite into a more dramatic or sales-heavy tone unless asked  
- overcomplicate clear rough drafts  
- change the intended audience without being told  
- introduce claims stronger than the input supports  

If a fact is unclear, the AI should keep the wording neutral or flag the gap.

---

## Human guidance for drafting rough scripts

When writing a rough script for AI refinement:

- focus on meaning, not polish  
- write in the order the story should be told  
- include what the viewer needs to understand  
- note what is shown visually  
- note what is happening behind the scenes only if it matters  

Do not worry about:

- elegance  
- repetition  
- wording quality  
- perfect flow  

The AI’s job is to fix those.
